## Executive Summary

The SQL tool data flow has **3 critical architectural issues**:

1. **❌ SCHEMA MISMATCH**: Tool queries `analytics_silver` (legacy) instead of `analytics_v2.fact_orders` (production)
2. **⚠️ RLS ENFORCEMENT**: RLS is set server-side but NOT enforced in SQL execution (connection doesn't use RLS)
3. **⚠️ CLIENT_ID EXPOSURE**: Though marked "never exposed to LLM", the prompt still includes client_id string in text (security smell)

### 6. RLS Context Setup (THE MAIN PROBLEM)
```python
with engine.connect() as conn:
    # Set RLS context in session
    conn.execute(
        sa_text("SELECT set_config('app.current_cliente_id', :cliente_id, false)"),
        {"cliente_id": str(real_client_id)},
    )
    conn.commit()  ← ⚠️ Committed but never used!

    # Execute query WITHOUT RLS enforcement
    cursor = conn.execute(sa_text(generated_sql))
    results = cursor.fetchall()
```

**CRITICAL ISSUE**:
1. RLS context is set via `set_config('app.current_cliente_id', ...)`
2. This sets the `app.current_cliente_id` variable
3. BUT the SQL query is executed on the **same connection** that set it
4. **However:** The query is raw SQL generated by LLM, not using RLS-aware ORM
5. **Result:** RLS policies are bypassed because:
   - RLS policies check `WHERE client_id = current_user_id OR client_id IN (SELECT id FROM clientes_vizu WHERE user_id = auth.uid())`
   - These policies are auth-based, not app-variable-based
   - The `app.current_cliente_id` setting is for the application, not PostgreSQL auth

**Status:** ❌ **BROKEN** - RLS not actually enforced

## 2. RLS Enforcement Analysis

### Current RLS Policies
From `20251130_add_rls_security.py`:

```sql
CREATE POLICY cliente_vizu_isolation ON cliente_vizu
    FOR ALL
    USING (id = COALESCE(
        current_setting('app.current_cliente_id', true)::uuid,
        '00000000-0000-0000-0000-000000000000'::uuid
    ));
```

**Problem:** This policy:
- ✅ Works with Supabase SDK (uses auth context)
- ❌ Does NOT work with SQLAlchemy raw connections
- ❌ The `app.current_cliente_id` setting is not the same as Supabase `auth.uid()`

### RLS in analytics tables
```sql
CREATE POLICY "analytics data isolation" ON analytics_silver
    FOR SELECT
    USING (
        client_id = current_setting('app.current_cliente_id')::text
    );
```

**Status:** ✅ This one SHOULD work because:
- It checks `client_id` directly
- Not auth-based
- Uses the `app.current_cliente_id` that was set

**BUT:** The `analytics_silver` table is LEGACY and should not be queried

## 4. The Core Problem

### User Has Data But SQL Returns None

**Why?**
1. User asks: "What is my revenue for the last 3 months?"
2. LLM generates: `SELECT SUM(total_revenue) FROM analytics_silver WHERE client_id = 'user-id' AND ...`
3. SQL executes and returns: **NOTHING** (analytics_silver is empty)
4. User sees: "No results found"
5. User's dashboard shows: Revenue = 0
6. User says: "But I can see the data in the dashboard!"

**The Real Issue:**
- Dashboard probably uses: `SELECT SUM(total_revenue) FROM analytics_v2.fact_order_metrics WHERE client_id = ... AND created_at > ...`
- SQL Tool tries: `SELECT ... FROM analytics_silver WHERE ...` (EMPTY TABLE)
- They're querying different tables!

# Add to sql_module.py

def validate_safe_sql_for_client(sql: str, client_id: str) -> bool:
    """
    Validates SQL for:
    - client_id filter present
    - Only analytics_v2 schema
    - No subqueries (for now)
    - No UPDATE/DELETE (already checked)
    """
    sql_upper = sql.upper()

    # Must contain client_id filter
